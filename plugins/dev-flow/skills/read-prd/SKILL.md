# 读 PRD · 方法论

## 你的任务

读取 PRD 目录中的所有文件（PDF、MD、HTML），提取需求要点，识别可独立交付的任务单元。

## 执行步骤

### 1. 文件扫描

用 Glob 扫描 PRD 目录：
- `**/*.pdf` — PDF 需求文档
- `**/*.md` — Markdown 需求文档
- `**/*.html`、`**/*.htm` — HTML 需求文档
- 忽略隐藏文件和 `.idea/` 等 IDE 目录

### 1.5 分配文件处理（`分配.md` 路由）

在 ben-workspace 的 PRD 结构中，批次目录下的 `分配.md` 是**索引文件**（列出本批次要做哪些 PRD），真实 PRD 在同项目上级的 `all/` 目录：

```
Prd/
└── 项目名/
    ├── all/              ← 真实 PRD 文件（唯一数据源）
    └── <person>_YYYYMMDD/ ← 批次目录
        └── 分配.md       ← 索引文件，指向 all/ 中的 PRD
```

**扫描时若命中 `分配.md`（或 `分配*.md`），按以下逻辑路由，不要把它当需求文档处理：**

1. **标记为分配文件**：文件名为 `分配.md` / `分配*.md` 时标记为「分配文件」，不直接作为 PRD 读取。
2. **解析分配文件内容**，支持两种形态：
   - 形态一（任务描述）：自由文本描述要做什么，如「完善 ben-plugins，记录个人标识……」
   - 形态二（文件名列表）：编号列出文件名，如「1. xxx功能.md / 2. yyy需求.pdf」
3. **定位 `all/` 目录**：相对批次目录向上找同项目的 `../all/`。
4. **在 `all/` 下匹配真实 PRD**：
   - 形态二：按列出的文件名精确匹配；不完全匹配时用文件名关键词模糊匹配。
   - 形态一：用任务描述中的关键词在 `all/` 下模糊匹配候选文件。
5. **匹配结果处理**：
   - 找到 → 按正常流程（第 2 步）读取匹配到的真实 PRD 文件。
   - 找不到 / 不确定 → **列出 `all/` 下所有文件**，用 AskUserQuestion 让用户手动指认对应 PRD。

**向下兼容**：若批次目录直接放了 PRD 文件（没有 `分配.md`，或没有 `all/` 结构），走原有扫描逻辑，行为不变。

### 2. 逐文件阅读

**MD 文件**：直接 Read 全部内容

**HTML 文件**：直接 Read 全部内容（HTML 本质是纯文本）
- 读取时忽略标签、样式、脚本噪音，聚焦正文需求内容
- 无需转换或清洗工具，Read 原文即可

**PDF 文件**：使用 Read 工具的 `pages` 参数分页读取
- 先读第 1 页，判断是否相关
- 相关则继续读后续页
- 每次最多读 10 页

### 3. 信息提取

对每个文件提取：

| 提取项 | 说明 |
|--------|------|
| 任务名称 | 简短描述（10 字以内） |
| 任务 ID | 如 ID1019077（从文件名或内容中提取） |
| 核心需求 | 这个任务要做什么（1-3 句话） |
| 涉及页面/模块 | 前端页面、后端模型、数据库表 |
| 关键约束 | 边界条件、依赖关系、特殊规则 |
| UI 要求 | 原型、配色、字段顺序等（如有） |

### 4. 需求摘要输出

按以下格式输出摘要，向用户确认：

```markdown
## 需求摘要

共发现 N 个文件，识别 M 个可交付任务：

### 任务 1：<任务名>（IDxxxxxxx）
- **核心需求**：...
- **涉及模块**：...
- **关键约束**：...

### 任务 2：<任务名>（IDxxxxxxx）
- **核心需求**：...
- **涉及模块**：...
- **关键约束**：...

...

请确认以上理解是否正确，或补充遗漏的需求。
```

## 注意事项

- PDF 可能包含扫描图片，Read 工具能处理大部分情况
- 如果 PDF 无法读取，提示用户转换为 MD 或手动粘贴内容
- HTML 可能夹带大量样式/脚本标签，读取时聚焦正文，忽略无关标记
- **Word（.doc/.docx）等二进制格式不支持**：Read 工具无法直接解析，遇到时提示用户先另存为 PDF/MD 再运行
- 不要遗漏任何文件，即使文件名看起来不相关
- 有些 MD 文件可能是总纲或索引，需要读完后关联到具体任务
